▸ Route Architecture — Replace the flat routes array

Your current app.routes.ts has 3 routes with a single guard. The new structure needs role-based lazy modules so the PATIENT user never downloads ADMIN code.

export const routes: Routes = [ { path: '', redirectTo: 'auth/login', pathMatch: 'full' }, { path: 'auth', loadChildren: () => import('./features/auth/auth.routes') }, { path: 'admin', loadChildren: () => import('./features/admin/admin.routes'), canActivate: [authGuard], canMatch: [roleGuard('ADMIN')] }, { path: 'doctor', loadChildren: () => import('./features/doctor/doctor.routes'), canActivate: [authGuard], canMatch: [roleGuard('DOCTOR')] }, { path: 'patient', loadChildren: () => import('./features/patient/patient.routes'), canActivate: [authGuard], canMatch: [roleGuard('PATIENT')] }, { path: '**', redirectTo: 'auth/login' } ];

⚡ patient-service — Build this first, everything depends on it

Your appointment-service has a PatientClient Feign interface calling patient-service at /patients/{id}. Until this service exists, appointments can't fully resolve patient data. This is the highest-priority backend gap.

// appointment-service already has this Feign client pointing at it @FeignClient(name = "patient-service") public interface PatientClient { @GetMapping("/patients/{id}") PatientDTO getPatient(@PathVariable "id" Long id); } // patient-service Entity needs at minimum: // id, firstName, lastName, dob, gender, bloodGroup, // phone, email, address, emergencyContact, // insuranceProvider, insurancePolicyNumber
🔧 api-gateway — Add routes for all new services + WebSocket support

Add WebSocket proxy support in Spring Cloud Gateway for the notification real-time channel, and add predicates for each new service path.

# application.yml additions spring.cloud.gateway.routes: - id: patient-service uri: lb://patient-service predicates: [Path=/patients/**] - id: medical-records-service uri: lb://medical-records-service predicates: [Path=/records/**, /diagnoses/**] - id: pharmacy-service uri: lb://pharmacy-service predicates: [Path=/prescriptions/**, /medications/**] - id: lab-service uri: lb://lab-service predicates: [Path=/labs/**, /lab-results/**] - id: billing-service uri: lb://billing-service predicates: [Path=/invoices/**, /payments/**] - id: notification-ws uri: lb:ws://notification-service predicates: [Path=/ws/**]
🔧 auth-service — Add token refresh + role-to-redirect handshake

Add a /auth/refresh endpoint and include the user's role in the JWT response so the frontend can route to the correct dashboard on login without a separate profile fetch.

// AuthResponse DTO — add role public record AuthResponse( String token, String refreshToken, // ADD String role, // ADD — ADMIN | DOCTOR | PATIENT Long expiresIn // ADD ) {}
⚡ notification-service — Event-driven via Kafka + WebSocket broadcast

All other services publish events to Kafka topics. Notification service consumes them and pushes to WebSocket clients or sends email/SMS via SendGrid/Twilio. Frontend connects to ws://api/ws with the JWT token.

// Topics consumed by notification-service appointment.created → reminder email 24h before + SMS appointment.cancelled → immediate SMS + email lab-result.ready → push notification + email invoice.generated → email with PDF attachment prescription.issued → pharmacy pickup SMS
